Finding the Port for SQL Server Express (Named Instances)
SQLEXPRESS uses a dynamic port — not 1433

SQL Server Express installs as a named instance (e.g., HOSTNAME\SQLEXPRESS) and listens on a random TCP port assigned at install time. DB Hub needs the actual port number, not the instance name.

Step 1 — Find the port via PowerShell

# Run this on the SQL Server machine Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Microsoft SQL Server\MSSQL16.SQLEXPRESS\MSSQLServer\SuperSocketNetLib\Tcp\IPAll" | Select-Object TcpDynamicPorts
Example output: TcpDynamicPorts: 64950
Use this number as the Port in DB Hub.

Step 2 — Make sure TCP/IP is enabled

If the key is empty or you get "connection refused":

  • Open SQL Server Configuration Manager
  • SQL Server Network Configuration → Protocols for SQLEXPRESS
  • Right-click TCP/IP → Enable
  • Restart the SQL Server service

Testing a Connection
Verify network path and credentials before saving

The Test Connection button opens a temporary connection to the server without saving anything. It returns the exact latency in milliseconds and the full error message if it fails.

Successful Test
Connected · 4ms latency
  • Host resolves and TCP port is reachable
  • Credentials are valid
  • Database name exists and account can access it
  • SSL negotiation succeeded (if enabled)
Failed Test
Connection refused (port 1433)
  • Check host IP and port number
  • Verify firewall allows DB Hub server → DB port
  • Ensure SQL Server TCP/IP is enabled
  • Confirm database name spelling
After saving, you can test any existing connection from its card on the Connections page using the Test Connection button. The result (✓ / ✗ and timestamp) is stored and shown on the card.

What to tell your security team: "DB Hub is a self-hosted internal web app that runs on our own server. User browsers connect only to DB Hub on port 8000/5173 — they never directly access database ports. The DB Hub server is the only machine that opens connections to PostgreSQL, SQL Server, and MySQL. You only need to allow DB Hub's server IP through to those DB ports, which is the same as giving any application server access."

Creating a DB Hub Service Account
Minimal-privilege read accounts for production databases
Never use a superuser/root/sa account in production. Create a dedicated dbhub_svc account with only the permissions DB Hub needs: read on tables and catalog views.

PostgreSQL — Create dbhub_svc

-- 1. Create login CREATE USER dbhub_svc WITH PASSWORD 'StrongPass123!'; -- 2. Grant read access on all tables GRANT CONNECT ON DATABASE shopdb TO dbhub_svc; GRANT USAGE ON SCHEMA public TO dbhub_svc; GRANT SELECT ON ALL TABLES IN SCHEMA public TO dbhub_svc; ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO dbhub_svc; -- 3. Grant catalog access for Schema Browser & Index Advisor GRANT SELECT ON pg_stat_user_tables, pg_stat_user_indexes, pg_indexes, pg_stat_statements TO dbhub_svc;

SQL Server — Create dbhub_svc (SQL Auth)

-- 1. Create server login CREATE LOGIN dbhub_svc WITH PASSWORD = 'StrongPass123!'; -- 2. Create user in target database USE AnalyticsDB; CREATE USER dbhub_svc FOR LOGIN dbhub_svc; -- 3. Read-only access EXEC sp_addrolemember 'db_datareader', 'dbhub_svc'; -- 4. Grant DMV access for Slow Query & Index Advisor GRANT VIEW SERVER STATE TO dbhub_svc; GRANT VIEW DATABASE STATE TO dbhub_svc;

MySQL — Create dbhub_svc

-- Replace 10.0.1.10 with your DB Hub server's IP CREATE USER 'dbhub_svc'@'10.0.1.10' IDENTIFIED BY 'StrongPass123!'; -- Read access on target schema GRANT SELECT ON testdb.* TO 'dbhub_svc'@'10.0.1.10'; -- Grant performance_schema access for Slow Query module GRANT SELECT ON performance_schema.* TO 'dbhub_svc'@'10.0.1.10'; FLUSH PRIVILEGES;

Troubleshooting Failed Connections
Common errors and how to fix them
"Connection refused" or "could not connect to server"
The DB Hub server cannot reach the database host on that port. Check: (1) is the host IP correct? (2) is the database service running? (3) does the firewall allow DB Hub server → database port? Run Test-NetConnection 10.0.1.20 -Port 5432 from the DB Hub server to diagnose.
SQL Server: "Connection refused on port 1433"
SQLEXPRESS uses a dynamic port (not 1433). Find the port: Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Microsoft SQL Server\MSSQL16.SQLEXPRESS\MSSQLServer\SuperSocketNetLib\Tcp\IPAll" | Select TcpDynamicPorts. Use that port number in the Port field. Also ensure TCP/IP is enabled in SQL Server Configuration Manager.
"password authentication failed for user"
The credentials are wrong, or the user doesn't exist on that database. Verify you can log in with the same user/password from another tool (psql, SSMS, MySQL Workbench). For PostgreSQL, check pg_hba.conf allows connections from the DB Hub server's IP.
SQL Server Windows Auth: "Login failed for user ''"
Windows Auth uses the identity of the process running DB Hub (the uvicorn/service account). That account must have a SQL Server login with access to the database. Either grant that account access, or switch to SQL Auth with a dedicated login.
MySQL: "Host '...' is not allowed to connect to this MySQL server"
The MySQL user was created with a specific host restriction. Use CREATE USER 'user'@'%' to allow any host, or specify the exact DB Hub server IP: 'user'@'10.0.1.10'. Run FLUSH PRIVILEGES after changing grants.
All modules work but Slow Query shows no data
The account needs access to monitoring views. For PostgreSQL: GRANT SELECT ON pg_stat_statements TO dbhub_svc. For SQL Server: GRANT VIEW SERVER STATE TO dbhub_svc. For MySQL: GRANT SELECT ON performance_schema.* TO dbhub_svc.
DB Hub shows "401 Unauthorized" on API calls
Your session token has expired (24-hour lifetime). Log out and log back in. This is a DB Hub session issue, not a database credential issue.
