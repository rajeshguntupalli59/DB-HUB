NOTE

SQL Server named instance requires dynamic port

SQL Server Express (SQLEXPRESS) uses a dynamic TCP port — not 1433. The actual port was found via registry lookup: TcpDynamicPorts = 64950. Using localhost\SQLEXPRESS as the host fails; using localhost with port 64950 succeeds.

How DB Hub helps: The Connections page shows a tip for named instances: "For named instances (SQLEXPRESS), use host + actual TCP port." This prevents a common misconfiguration that causes hours of debugging.

FINDING

Real problem caught: orders.customer_id has no index

The schema browser showed that orders.customer_id is a foreign key column with no supporting index. Every JOIN from orders to customers does a full scan of the orders table. With 2,000 rows this is acceptable; at 200,000 rows it becomes a bottleneck.

How DB Hub helps: Schema Browser exposes the raw index list per table. A developer can instantly see which FK columns have no index without writing catalog queries by hand.

REAL SCENARIO

How this prevents production incidents

In a typical deploy workflow: take snapshot before migration → run migration → take snapshot after → diff. If the diff shows an unexpected column drop or type change, the deploy can be rolled back before users are affected. Without Change Tracker, teams rely on reading migration files manually — which misses the actual state of the database if a migration partially failed.

TC-7.2 — Generates accurate SQL using real column names

Question:

"Write a query to find the top 5 customers by total order value"

-- Generated by AI Assistant (uses real schema column names):
SELECT c.id, c.first_name, c.last_name, c.email,
       SUM(oi.quantity * oi.unit_price) AS total_value
FROM customers c
JOIN orders o ON o.customer_id = c.id
JOIN order_items oi ON oi.order_id = o.id
WHERE c.is_active = true
GROUP BY c.id, c.first_name, c.last_name, c.email
ORDER BY total_value DESC
LIMIT 5;

Result: Used first_name, last_name, unit_price, is_active — all actual column names from the live schema. The query is syntactically correct and would run without modification.

TC-8.2 — Index Advisor finds 4 real missing indexes on shopdb

POST /api/optimizer/3/indexes
{"sql":"SELECT * FROM orders o JOIN customers c ON o.customer_id=c.id WHERE c.country='US' ORDER BY o.created_at"}

Recommendations:
  [Medium-High] orders.customer_id
    Reason: FK column with no index — causes slow joins when filtering orders by customer
    DDL: CREATE INDEX idx_orders_customer_id ON orders (customer_id);

  [Medium-High] order_items.order_id
    Reason: FK column with no index — slow when loading items for a specific order
    DDL: CREATE INDEX idx_order_items_order_id ON order_items (order_id);

  [High] customers.country
    Reason: WHERE clause filter column — full table scan on every country query
    DDL: CREATE INDEX idx_customers_country ON customers (country);

    Reason: ORDER BY column — sorts require seq scan without this index
    DDL: CREATE INDEX idx_orders_created_at ON orders (created_at);

These are real production problems. All 4 missing indexes were verified by checking pg_indexes. The shopdb test database had only primary key indexes — no indexes on any FK or filter columns. On a table with millions of rows, these gaps would cause query times in the seconds range. DB Hub found them in under 1 second.

Issues Found & Fixed During Testing

BUG FIXED

MySQL: rows is a reserved word in MySQL 8.4

The slow query SQL used SUM_ROWS_SENT AS rows — but rows is a reserved keyword in MySQL 8.4, causing a syntax error. Renamed to row_count in the backend. Frontend updated to accept both field names for backward compatibility.

Fixed in: services/optimizer_slow.py

BUG FIXED

OptimizerPage was reading wrong localStorage key

The OptimizerPage frontend was calling localStorage.getItem("db_hub_token") but the auth system stores the token under "token". All API calls from the optimizer would silently fail in the browser with 401 errors.

Fixed in: frontend/src/pages/OptimizerPage.jsx

BUG FIXED

MySQL/SQL Server code sharing via else: instead of elif

In erd.py, tracker.py, and assistant.py the branch was: if postgresql: ... else: # SQL Server. MySQL connections fell into the SQL Server branch, causing errors like "Table 'sys.foreign_keys' doesn't exist" on MySQL. Fixed to elif db_type == "sqlserver" with a separate if db_type == "mysql" block.

Fixed in: routers/erd.py, routers/tracker.py, routers/assistant.py

BUG FIXED

SQL Server: STRING_AGG wrong syntax + tds_version blocking modern T-SQL

The tracker used MySQL syntax STRING_AGG(col, ',' ORDER BY ...) instead of T-SQL syntax STRING_AGG(col, ',') WITHIN GROUP (ORDER BY ...). Additionally, tds_version="7.0" in the pymssql connection was blocking modern T-SQL features. Both removed.

Fixed in: routers/tracker.py, db_connect.py, routers/connections.py

Real Database Problems DB Hub Caught

These are genuine issues found in the test database during testing — not simulated. They represent the kind of problems DB Hub surfaces in real production environments.

Problem 1: FK columns with no indexes — every join is a full table scan

The shopdb test database had orders.customer_id and order_items.order_id as foreign key columns with zero indexes. Any JOIN involving these columns scans the entire table. At production scale (millions of rows), this means multi-second query times on every page load that shows order history.

→

Caught by: Index Advisor (Tab 2 of Query Optimizer). Found in 1 second, generated ready-to-run CREATE INDEX DDL.

Problem 2: Anti-pattern SQL in active queries — WHERE LOWER(status) = 'pending'

A common pattern for case-insensitive matching: wrap the column in LOWER(). The problem is this disables any index on the status column entirely. The database must apply the function to every row before filtering. The correct fix is either storing lowercase consistently, or creating a function-based index.

→

Caught by: SQL Rewriter — "Function on indexed column" rule (HIGH severity). Explanation and alternative approach included.

Problem 3: NOT IN (SELECT ...) returns zero rows silently when subquery has NULLs

A query using WHERE id NOT IN (SELECT cancelled_id FROM cancellations) will return zero rows if the cancellations table contains even one NULL. This is one of SQL's most dangerous silent bugs — the query runs without error but returns wrong results. Swapping to NOT EXISTS fixes the NULL behavior.

→

Caught by: SQL Rewriter — "NOT IN with subquery" rule (MEDIUM severity). Auto-generates the NOT EXISTS equivalent.
