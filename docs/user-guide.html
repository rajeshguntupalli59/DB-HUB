DB Hub User Guide

Version 1.0 · Updated May 2026

What is DB Hub?

DB Hub is a self-hosted database management platform that gives your entire team a shared workspace for everything database-related. Instead of juggling five different tools, DB Hub puts schema browsing, documentation, ER diagrams, query optimization, change tracking, and AI assistance in one application — connected to your PostgreSQL, SQL Server, and MySQL databases.

DB Hub runs entirely on your own server. No cloud sync, no usage tracking, no vendor lock-in. Your database credentials and schema data never leave your network.

  • · Connection Manager
  • · Schema Browser

Requirements

Server requirements

  • Python 3.11 or higher — check with python --version
  • 500 MB free disk space (for the app + SQLite database)
  • Any OS: Windows 10+, Ubuntu 20.04+, macOS 12+
  • Network access to your database servers from the DB Hub host

Browser requirements

  • Any modern browser: Chrome, Firefox, Edge, Safari (2022+)
  • JavaScript must be enabled

Optional — for AI features only

  • An Anthropic API key (get one at console.anthropic.com) — needed for AI Documentation and AI Assistant modules only
  • The other 6 modules work with no API key at all

Note: DB Hub creates its own SQLite database file on first run. You do not need to install or configure a separate database server to run DB Hub itself.

Installation

Step 1 — Download and extract

After purchase you will receive a ZIP file. Extract it to a permanent location on your server — for example C:\db-hub\ on Windows or /opt/db-hub/ on Linux.

Step 2 — Create the environment file

Inside the backend/ folder, create a file named .env with the following content:

# Required
SECRET_KEY=replace-with-a-random-64-character-string
ENCRYPTION_KEY=replace-with-a-fernet-key

# Optional — only needed for AI Documentation and AI Assistant
ANTHROPIC_API_KEY=sk-ant-...

Generate your keys with Python:

python -c "import secrets; print(secrets.token_hex(32))"
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"

Important: Keep these keys safe. If you lose the ENCRYPTION_KEY, saved database passwords cannot be decrypted. Back up the .env file separately from the application.

Step 3 — Install Python dependencies

cd backend
pip install -r requirements.txt

Step 4 — Start DB Hub

Windows (PowerShell)

cd backend
python -m uvicorn main:app --host 0.0.0.0 --port 8000

Linux / macOS

cd backend
python -m uvicorn main:app --host 0.0.0.0 --port 8000

Running as a background service (Linux — systemd)

[Unit]
Description=DB Hub
After=network.target

[Service]
WorkingDirectory=/opt/db-hub/backend
ExecStart=/usr/bin/python3 -m uvicorn main:app --host 0.0.0.0 --port 8000
Restart=always

[Install]
WantedBy=multi-user.target

Save to /etc/systemd/system/dbhub.service, then run systemctl enable --now dbhub.

Step 5 — Open DB Hub

Navigate to http://your-server-ip:8000 in your browser. On the same machine, use http://localhost:8000.

First Login

The first time you open DB Hub, you will see a Setup screen instead of a login form. This is how the first admin account is created.

1

3

The setup screen only appears once. After the first admin is created, it redirects to the login page permanently. To create more users, use the Team module after logging in.

Adding a Database Connection

All modules in DB Hub share the same saved connections. You configure a connection once, and every module can use it.

1

Click Connections in the left sidebar.

2

Click New Connection in the top right.

3

Choose your database type, fill in the host, port, database name, and credentials.

4

Click Test Connection to verify. Green means you're ready.

5

Click Save. The connection is now available in all modules.

Passwords are encrypted with AES-128 (Fernet) before being stored. They are never stored in plain text.

Named instances (e.g. SQLEXPRESS)

SQL Server Express and named instances do not use port 1433 by default — they use a dynamic TCP port. To find it, run this on the SQL Server machine:

(Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Microsoft SQL Server\MSSQL16.SQLEXPRESS\MSSQLServer\SuperSocketNetLib\Tcp\IPAll").TcpDynamicPorts

Use that number as the Port in DB Hub. Hostname stays as the machine IP — do not put localhost\SQLEXPRESS in the host field.

Windows Auth tip: DB Hub must be running on a Windows machine that is joined to the same domain (or the same local machine) as the SQL Server for Windows Authentication to work.

Module 1 · Team & Access

The Team module lets you invite colleagues and control what they can see and do inside DB Hub.

Roles

admin

Full access. Can create users, delete connections, manage everything. Only admins can access the Team module.

editor

Can use all 8 modules, add connections, generate docs, run optimizer queries, and take snapshots.

viewer

Read-only. Can browse schemas, view docs, view ERDs, and view snapshots. Cannot add connections or run write operations.

Adding a team member

  1. Go to Team in the sidebar (admin only).
  2. Click Invite User.
  3. Enter their name, email, and choose a role.
  4. Set a temporary password — they can change it after first login.
  5. Share the DB Hub URL and their credentials with them.

Module 2 · Schema Browser

Browse every table, column, index, and foreign key in your database in real time. No caching — the Schema Browser always reflects the live state of your database.

How to use it

  1. Click Schema Browser in the sidebar.
  2. Pick a connection from the dropdown at the top.
  3. The left panel lists all schemas and tables. Click any table to open it.
  4. The right panel shows Columns, Indexes, and Foreign Keys tabs.

What you can see

  • Columns — name, data type, nullable, default value, primary key flag
  • Indexes — index name, type (btree/hash/etc.), columns covered, uniqueness
  • Foreign Keys — source column, target table, target column, constraint name

Tip: Use Schema Browser alongside the Query Optimizer — find the table structure here, then paste a query into the optimizer to analyze it.

Module 3 · AI Documentation

Generate professional markdown documentation for any table using Claude AI. The AI reads your column names, types, constraints, and foreign keys to write a description that actually makes sense.

Requires: ANTHROPIC_API_KEY in your .env file. AI calls are billed directly to your Anthropic account at standard rates.

Generating docs for a table

  1. Click Documentation in the sidebar.
  2. Select a connection and then a table from the dropdowns.
  3. Click Generate Docs. This takes 5–15 seconds.
  4. The doc appears in the right panel in formatted markdown.
  5. Edit the doc if needed — it's fully editable after generation.
  6. Click Save to persist it.

Exporting docs

Click Export All as Markdown to download a single .md file containing documentation for every table that has been generated. This file can be dropped directly into GitHub, Notion, Confluence, or any markdown-compatible system.

Tip: Generate docs once, then maintain them manually. The AI gives you a solid first draft — your team adds the business context on top.

Module 4 · ER Diagram

An interactive visual map of your schema. Tables are displayed as draggable cards; foreign key relationships are drawn as connecting lines. The layout is automatic — drag tables to arrange them however makes sense for your team.

How to use it

  1. Click ER Diagram in the sidebar.
  2. Select a connection. The diagram loads automatically.
  3. Drag any table card to rearrange. The relationship lines follow.
  4. Each table card shows all columns and their types.
  5. FK lines show the source column → target table relationship.

Reading the diagram

  • Table cards — table name at the top, columns listed below with type
  • Connecting lines — each line represents a foreign key. The line starts at the FK column and ends at the referenced table.
  • Pan and zoom — use scroll to zoom, click-drag on the background to pan

Tip: The ER Diagram is great for onboarding new team members. Take a screenshot of the arranged diagram and add it to your project wiki.

Module 5 · Change Tracker

The Change Tracker takes a snapshot of your entire schema and lets you diff any two snapshots to see exactly what changed — new tables, dropped columns, added indexes, modified types.

Taking a snapshot

  1. Click Change Tracker in the sidebar.
  2. Select a connection.
  3. Click Take Snapshot. DB Hub records the full schema at this moment.
  4. Give the snapshot a label (e.g. "Before v2.3 deploy") — or leave it blank for an auto-generated timestamp.

Comparing snapshots

  1. With at least two snapshots taken, click Compare.
  2. Select a "before" snapshot and an "after" snapshot from the dropdowns.
  3. The diff view shows three categories:
    • Added — new tables or columns
    • Removed — dropped tables or columns
    • Modified — type changes, nullability changes, new indexes

Recommended workflow

  • Snapshot before every migration or deploy
  • Snapshot after the migration completes
  • Use the diff to confirm the migration applied exactly the expected changes

Important: Snapshots capture schema structure only — not data. They are safe to take on production databases.

Module 6 · AI Assistant

A schema-aware AI chat assistant. It reads your live table structure before answering so its suggestions are specific to your actual column names, types, and relationships — not generic SQL advice.

Requires: ANTHROPIC_API_KEY in your .env file.

How to use it

  1. Click AI Assistant in the sidebar.
  2. Select a connection — the assistant automatically loads your schema.
  3. Type your question in the input box and press Enter or click Send.

What to ask it

  • "What indexes do I have on the orders table?"
  • "Write a query to find all customers who placed more than 5 orders last month."
  • "Why might my users table be slow to query?"
  • "What does the status column in the invoices table likely represent?"
  • "Which tables have foreign keys pointing to the products table?"

Tip: The assistant remembers the conversation within a session. You can ask follow-up questions — "now rewrite that query to avoid the subquery" — and it keeps context.

Module 7 · Query Optimizer

Four performance tools in one tab. Paste a SQL query or select a connection to identify and fix performance problems across PostgreSQL, SQL Server, and MySQL.

Tab 1 — EXPLAIN Visualizer

Visualizes the execution plan for any SQL query as an interactive tree. Each node shows cost, estimated rows, and the operation type. Warnings are highlighted automatically.

  1. Select a connection and click the EXPLAIN tab.
  2. Paste your SQL query into the editor.
  3. Optional: enable ANALYZE (actually runs the query and shows real timing — PostgreSQL only) or BUFFERS (shows cache hit/miss stats — PostgreSQL only).
  4. Click Run EXPLAIN.
  5. The plan tree appears below. Expand nodes to see child operations.
  6. The Summary bar shows total cost, plan rows, and node type at a glance.
  7. Warnings (if any) appear below the summary — e.g. "Sequential scan on large table" or "Hash join with high row estimate error."

ANALYZE mode runs the query. Do not use ANALYZE on INSERT/UPDATE/DELETE statements in production — use it only on SELECT queries.

Tab 2 — Index Advisor

Recommends missing indexes based on your query's WHERE and ORDER BY clauses, and by checking for foreign key columns without a supporting index in the live catalog.

  1. Select a connection and click the Index Advisor tab.
  2. Paste the query you want to optimize.
  3. Click Analyze.
  4. Each recommendation card shows:
    • The table and column(s) to index
    • The reason (WHERE clause, ORDER BY, missing FK index)
    • The estimated benefit (High / Medium / Low)
    • A ready-to-run CREATE INDEX statement — click Copy to grab it

Tip: Review recommendations before applying them. More indexes improve reads but slow writes. For OLTP tables with heavy INSERT/UPDATE workloads, prioritize only High-benefit recommendations.

Tab 3 — Slow Query Dashboard

Shows your historically slowest queries, sorted by mean execution time. Data comes directly from your database's built-in statistics — no agents or instrumentation required.

PostgreSQL

Reads from pg_stat_statements. Enable with: CREATE EXTENSION pg_stat_statements;

SQL Server

Reads from sys.dm_exec_query_stats and sys.dm_exec_sql_text. No setup required — DMVs are always available.

MySQL

Reads from performance_schema.events_statements_summary_by_digest. Enabled by default in MySQL 8.0+.

Columns shown: Query text, Calls, Mean time (ms), Total time, Min/Max time, Rows returned, Cache hit % (PostgreSQL only).

Tab 4 — SQL Rewriter

Paste any SQL query and DB Hub checks it against 8 common anti-pattern rules. No database connection needed — this analysis is pure text.

High
  • · SELECT * — fetch all columns
  • · OR in WHERE — prevents index use
  • · Function on indexed column
Medium
  • · LIKE '%value' — leading wildcard
  • · SELECT DISTINCT — join problem
  • · NOT IN (subquery) — NULL unsafe
  • · Implicit cross join (FROM a, b)
Low
  • · Large IN list (>50 literals)
  • · Large OFFSET pagination

Each finding shows the issue, a plain-English explanation, a suggested fix, and — where possible — an auto-rewritten version of the SQL. Click Copy to grab the rewrite.

Enterprise Deployment

How DB Hub fits inside a corporate network

The most common question from enterprise buyers is: "Our firewall blocks external tools from reaching our databases — how does DB Hub work?"

DB Hub is not an external tool. You install it on a server inside your own network. Users access it through a web browser on port 8000 (or 443 with a reverse proxy). The browser never touches a database port. Only the DB Hub server makes database connections — and that server is already inside your perimeter.

// How traffic actually flows

[User Browser]

│ HTTP port 8000 (or HTTPS 443)

│ ← only this crosses any desk/VPN boundary

▼

[DB Hub Server] ← runs on your internal server

│ TCP 5432 (PostgreSQL)

│ TCP 1433 (SQL Server)

│ TCP 3306 (MySQL)

│ ← internal server-to-server only

▼

[Database Servers] ← never exposed to browsers

// The only external call (optional):

DB Hub → api.anthropic.com (AI features only, uses your key)

This means your database servers never need new firewall rules for browser clients. The only rule needed is the one allowing DB Hub server → database server on the database port — the same type of rule you would add for any internal application server.

How to pitch this to your security team: "DB Hub is a self-hosted internal web application. It's equivalent to deploying an internal wiki or monitoring tool. The database ports are only accessed server-to-server, not from user workstations."

Firewall rules required

This is the complete list of network rules needed to run DB Hub in a corporate environment. Nothing else.

Source Destination Port Why
User workstations / VPN DB Hub server 8000 (or 443) Browser access to the web UI
DB Hub server PostgreSQL server 5432 DB Hub reading schema & stats
DB Hub server SQL Server 1433 (or dynamic) DB Hub reading schema & stats
DB Hub server MySQL server 3306 DB Hub reading schema & stats
DB Hub server (optional) api.anthropic.com 443 AI features only — can be blocked if AI not used

No outbound database ports from user workstations. Users only need HTTP access to the DB Hub server. This is why DB Hub passes security reviews that block tools like DBeaver or DataGrip, which require direct database port access from each developer's laptop.

Running DB Hub as a Windows domain service account

For SQL Server environments using Windows Authentication, DB Hub needs to run under a domain account that SQL Server trusts. Here is the recommended setup.

Step 1 — Create the service account

Ask your AD admin to create a domain service account, for example CORP\dbhub-svc. It does not need any special AD rights — just a standard domain user account with a non-expiring password.

Step 2 — Grant SQL Server access to the service account

Run the following on each SQL Server that DB Hub will connect to (see the full privilege scripts in the next sections):

-- Create login for the service account
CREATE LOGIN [CORP\dbhub-svc] FROM WINDOWS;

-- Grant read access to specific databases
USE YourDatabase;
CREATE USER [CORP\dbhub-svc] FOR LOGIN [CORP\dbhub-svc];
ALTER ROLE db_datareader ADD MEMBER [CORP\dbhub-svc];

-- Grant VIEW SERVER STATE for slow query dashboard
GRANT VIEW SERVER STATE TO [CORP\dbhub-svc];

Step 3 — Run DB Hub under the service account

Option A — Windows Service (recommended for production)

Install DB Hub as a Windows Service using NSSM (Non-Sucking Service Manager):

# Download nssm from nssm.cc, then:
nssm install DBHub "C:\Python311\python.exe"
nssm set DBHub AppParameters "-m uvicorn main:app --host 0.0.0.0 --port 8000"
nssm set DBHub AppDirectory "C:\db-hub\backend"
nssm set DBHub ObjectName "CORP\dbhub-svc" "ServiceAccountPassword"
nssm set DBHub Start SERVICE_AUTO_START
nssm start DBHub

Option B — Scheduled Task (simpler)

Create a Scheduled Task in Windows Task Scheduler:

  • Action: python -m uvicorn main:app --host 0.0.0.0 --port 8000
  • Start in: C:\db-hub\backend
  • Run as: CORP\dbhub-svc
  • Trigger: At system startup
  • Check "Run whether user is logged in or not"

Identity note: When DB Hub connects to SQL Server using Windows Auth (blank username/password in the connection), SQL Server sees the identity of the process — CORP\dbhub-svc. All DB Hub users share this identity when connecting to SQL Server. Use per-user SQL credentials if you need per-user SQL Server audit trails.

Running on Linux with Kerberos (advanced)

If your DB Hub server is Linux but SQL Server uses Windows Auth, you can configure Kerberos authentication so the Linux server can authenticate to SQL Server as a domain account. This requires:

  • Joining the Linux server to the domain (using realm join or sssd)
  • Configuring /etc/krb5.conf with your domain's KDC
  • Using pymssql with Kerberos credentials

This is complex to set up. For most environments, running DB Hub on a Windows server and using a domain service account is the simpler path. Alternatively, create a SQL Auth login (see next section) and avoid Windows Auth entirely.

PostgreSQL — Minimal privilege setup

Create a dedicated read-only login for DB Hub. This account only needs the permissions DB Hub actually uses — no superuser, no write access.

-- ─────────────────────────────────────────────────────────
-- DB Hub — PostgreSQL minimal privilege setup
-- Run as a superuser (e.g. postgres)
-- Replace 'YourDatabase' with your actual database name
-- Replace 'StrongPassword123!' with a secure password
-- ─────────────────────────────────────────────────────────

-- 1. Create the login
CREATE USER dbhub_svc WITH PASSWORD 'StrongPassword123!';

-- 2. Allow connection to the target database
GRANT CONNECT ON DATABASE "YourDatabase" TO dbhub_svc;

-- 3. Allow schema inspection
\c YourDatabase
GRANT USAGE ON SCHEMA public TO dbhub_svc;

-- 4. Read-only on all current tables
GRANT SELECT ON ALL TABLES IN SCHEMA public TO dbhub_svc;

-- 5. Read-only on future tables automatically
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO dbhub_svc;

-- 6. Allow reading pg_stat_statements (slow query dashboard)
--    This view requires the extension to be installed first:
--    CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
GRANT SELECT ON pg_stat_statements TO dbhub_svc;

-- 7. Allow reading system catalog views (schema browser, ERD)
--    These are already readable by any login — no extra grant needed.

-- ─────────────────────────────────────────────────────────
-- What this account CAN do:
--   ✓ Browse tables, columns, indexes, foreign keys
--   ✓ View ER diagrams
--   ✓ Generate AI documentation (reads schema metadata)
--   ✓ View slow queries via pg_stat_statements
--   ✓ Run EXPLAIN (no ANALYZE — that executes the query)
--   ✓ Index advisor (reads catalog tables)
--
-- What this account CANNOT do:
--   ✗ INSERT / UPDATE / DELETE any data
--   ✗ Create or drop tables or indexes
--   ✗ Access other schemas (unless granted separately)
-- ─────────────────────────────────────────────────────────

Multiple schemas: If your database uses non-public schemas (e.g. sales, hr), repeat the GRANT USAGE and GRANT SELECT steps for each schema. DB Hub's schema browser will show all schemas the user has access to.

pg_hba.conf — allow the DB Hub server

Add a line to pg_hba.conf that allows dbhub_svc to connect from the DB Hub server's IP:

# TYPE  DATABASE        USER         ADDRESS            METHOD
host    YourDatabase    dbhub_svc    10.0.1.50/32       scram-sha-256
#                                    ^ DB Hub server IP

Then reload: SELECT pg_reload_conf();

SQL Server — Minimal privilege setup

Two options depending on your environment: SQL Authentication (works anywhere) or Windows Authentication (domain environments).

Option A — SQL Authentication login

-- ─────────────────────────────────────────────────────────
-- DB Hub — SQL Server minimal privilege setup (SQL Auth)
-- Run as sysadmin. Replace placeholders as needed.
-- ─────────────────────────────────────────────────────────

USE master;

-- 1. Create the login
CREATE LOGIN dbhub_svc
  WITH PASSWORD = 'StrongPassword123!',
       CHECK_EXPIRATION = OFF,
       CHECK_POLICY = ON;

-- 2. Grant VIEW SERVER STATE (needed for slow query dashboard)
--    This reads sys.dm_exec_query_stats and sys.dm_exec_sql_text
GRANT VIEW SERVER STATE TO dbhub_svc;

-- 3. Grant VIEW ANY DEFINITION (needed to read index & FK metadata)
GRANT VIEW ANY DEFINITION TO dbhub_svc;

-- 4. For each database DB Hub should access, run:
USE YourDatabase;
CREATE USER dbhub_svc FOR LOGIN dbhub_svc;

-- Read-only data access
ALTER ROLE db_datareader ADD MEMBER dbhub_svc;

-- ─────────────────────────────────────────────────────────
-- What this account CAN do:
--   ✓ Browse tables, columns, indexes, foreign keys (sys.*)
--   ✓ View ER diagrams
--   ✓ Generate AI documentation (reads sys.columns, sys.tables)
--   ✓ Slow query dashboard (sys.dm_exec_query_stats)
--   ✓ Missing index advisor (sys.dm_db_missing_index_details)
--   ✓ EXPLAIN via SET SHOWPLAN_ALL ON
--
-- What this account CANNOT do:
--   ✗ INSERT / UPDATE / DELETE any data
--   ✗ Create or drop objects
--   ✗ Access other databases (unless user created there too)
-- ─────────────────────────────────────────────────────────

Option B — Windows Authentication login (domain service account)

-- ─────────────────────────────────────────────────────────
-- DB Hub — SQL Server minimal privilege setup (Windows Auth)
-- Use when DB Hub runs as CORP\dbhub-svc on a domain machine
-- ─────────────────────────────────────────────────────────

USE master;

-- 1. Create login from Windows account
CREATE LOGIN [CORP\dbhub-svc] FROM WINDOWS;

-- 2. Same server-level grants as SQL Auth
GRANT VIEW SERVER STATE TO [CORP\dbhub-svc];
GRANT VIEW ANY DEFINITION TO [CORP\dbhub-svc];

-- 3. Per-database access
USE YourDatabase;
CREATE USER [CORP\dbhub-svc] FOR LOGIN [CORP\dbhub-svc];
ALTER ROLE db_datareader ADD MEMBER [CORP\dbhub-svc];

-- ─────────────────────────────────────────────────────────
-- In DB Hub connection settings:
--   Host     : 10.0.1.20   (SQL Server IP — not hostname\INSTANCE)
--   Port     : 1433         (or dynamic port for named instances)
--   Database : YourDatabase
--   Username : (leave blank)
--   Password : (leave blank)
-- ─────────────────────────────────────────────────────────

Named instance port: SQL Server Express and named instances use a dynamic TCP port instead of 1433. Find it with: (Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Microsoft SQL Server\MSSQL16.SQLEXPRESS\MSSQLServer\SuperSocketNetLib\Tcp\IPAll").TcpDynamicPorts

Verifying the grants work

-- Connect as dbhub_svc and run these to verify:
SELECT name FROM sys.tables;                         -- schema browser
SELECT * FROM sys.dm_exec_query_stats;              -- slow queries
SELECT * FROM sys.dm_db_missing_index_details;      -- index advisor
SET SHOWPLAN_ALL ON; SELECT 1; SET SHOWPLAN_ALL OFF; -- explain

MySQL — Minimal privilege setup

-- ─────────────────────────────────────────────────────────
-- DB Hub — MySQL minimal privilege setup
-- Run as root or a user with GRANT OPTION
-- Replace placeholders as needed
-- ─────────────────────────────────────────────────────────

-- 1. Create the user
--    Replace '10.0.1.50' with your DB Hub server's IP
--    Use '%' to allow from any host (less secure)
CREATE USER 'dbhub_svc'@'10.0.1.50' IDENTIFIED BY 'StrongPassword123!';

-- 2. Read-only access to your application database
GRANT SELECT ON YourDatabase.* TO 'dbhub_svc'@'10.0.1.50';

-- 3. Read information_schema (schema browser, ERD, index advisor)
--    information_schema is readable by all users — no extra grant needed

-- 4. Read performance_schema (slow query dashboard)
GRANT SELECT ON performance_schema.events_statements_summary_by_digest
  TO 'dbhub_svc'@'10.0.1.50';

-- 5. Apply changes
FLUSH PRIVILEGES;

-- ─────────────────────────────────────────────────────────
-- What this account CAN do:
--   ✓ Browse tables, columns, indexes, foreign keys
--   ✓ View ER diagrams
--   ✓ Generate AI documentation
--   ✓ Slow query dashboard (performance_schema)
--   ✓ Index advisor (information_schema.STATISTICS, KEY_COLUMN_USAGE)
--   ✓ EXPLAIN queries
--
-- What this account CANNOT do:
--   ✗ INSERT / UPDATE / DELETE any data
--   ✗ Create or drop tables or indexes
--   ✗ Access other databases
-- ─────────────────────────────────────────────────────────

Multiple databases: Repeat the GRANT SELECT statement for each database DB Hub should access. Use GRANT SELECT ON *.* TO 'dbhub_svc'@'10.0.1.50'; only if you want DB Hub to see all databases on the server.

Verify the grants work

-- Connect as dbhub_svc and run:
SHOW TABLES FROM YourDatabase;
SELECT TABLE_NAME FROM information_schema.TABLES WHERE TABLE_SCHEMA = 'YourDatabase';
SELECT DIGEST_TEXT, COUNT_STAR
  FROM performance_schema.events_statements_summary_by_digest LIMIT 5;

MySQL 8.0 auth note: If your application cannot connect after creating the user, it may be a caching_sha2_password vs mysql_native_password mismatch. Run: ALTER USER 'dbhub_svc'@'10.0.1.50' IDENTIFIED WITH mysql_native_password BY 'StrongPassword123!';

Troubleshooting

"Unable to connect" — PostgreSQL

  • Confirm PostgreSQL is running: pg_isready -h localhost -p 5432
  • Check pg_hba.conf allows connections from the DB Hub host
  • Make sure the database name is correct (not the server hostname)
  • Try connecting with psql using the same credentials to isolate the issue

"Unable to connect" — SQL Server

  • Ensure TCP/IP is enabled in SQL Server Configuration Manager
  • For named instances (SQLEXPRESS), use the dynamic port instead of 1433 — see the SQL Server setup section above
  • Windows Firewall may be blocking port — add an inbound rule for the port
  • For Windows Auth, DB Hub must run on the same Windows machine or domain

"Unable to connect" — MySQL

  • Check MySQL is running: mysqladmin -u root -p status
  • Ensure the user has remote access: GRANT ALL ON mydb.* TO 'user'@'%';
  • Check bind-address in my.cnf — set to 0.0.0.0 if connecting remotely

AI features not working

  • Confirm ANTHROPIC_API_KEY is set in backend/.env
  • Restart the backend after changing .env
  • Check your Anthropic account has remaining credits
  • The key should start with sk-ant-

Slow query dashboard shows no data — PostgreSQL

  • Run: CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
  • Add pg_stat_statements to shared_preload_libraries in postgresql.conf and restart
  • Execute some queries first — the view only shows queries run since the extension was enabled

Port 8000 already in use

# Windows
netstat -ano | findstr :8000
Stop-Process -Id <PID> -Force

# Linux
lsof -i :8000
kill -9 <PID>

Or start DB Hub on a different port: uvicorn main:app --port 8080

FAQ

Can I run DB Hub behind a reverse proxy (nginx)?

Yes. Point nginx to http://localhost:8000. Make sure to set proxy_set_header Host $host; and proxy_set_header X-Real-IP $remote_addr;. For WebSocket support (not currently required), add the appropriate upgrade headers.

Can I use HTTPS?

Run DB Hub behind nginx or Caddy with an SSL certificate. The FastAPI backend itself speaks plain HTTP — the reverse proxy handles TLS termination.

How do I back up DB Hub's data?

Copy the db_hub.sqlite3 file (in the backend/ folder by default) and your .env file. That's everything. Restore by placing them back in the same locations.

Can I connect databases that are on a different server?

Yes — DB Hub connects over the network. The database just needs to be reachable from the machine running DB Hub. Use the database server's IP or hostname in the Host field.

Does DB Hub modify my database?

No. DB Hub is read-only with respect to your databases. It reads schema metadata and query statistics — it never writes to your application databases. The only database it writes to is its own SQLite file.

What happens if I change my ENCRYPTION_KEY?

All stored database passwords become unreadable. Do not change the ENCRYPTION_KEY after the first run unless you plan to re-enter all connection passwords.

How many database connections can I store?

Unlimited. You can save as many PostgreSQL, SQL Server, and MySQL connections as your team needs — mix and match freely.
